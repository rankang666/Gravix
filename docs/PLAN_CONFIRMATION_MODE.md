# 计划确认模式 (Plan-First Mode)

## 概述

**计划确认模式**是一种"人在回路"(Human-in-the-Loop)的工具执行方式，强调用户对系统行为的完全控制。

## 核心理念

```
先规划 → 后确认 → 再执行
```

## 工作流程

```
┌─────────────────────────────────────────────────────────┐
│  1. 用户提问                                             │
└──────────────────┬──────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────────┐
│  2. LLM 生成执行计划                                      │
│     - 分析需求                                           │
│     - 确定需要的工具                                       │
│     - 规划执行步骤                                         │
└──────────────────┬──────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────────┐
│  3. 展示计划给用户                                         │
│     - 说明每一步的目的                                      │
│     - 显示精确的工具调用                                    │
│     - 描述预期结果                                         │
└──────────────────┬──────────────────────────────────────┘
                   ↓
            ┌──────┴──────┐
            │  用户决定    │
            └──────┬──────┘
         确认│          │拒绝/修改
            ↓           ↓
    ┌───────────┐  ┌──────────┐
    │ 4. 执行   │  │ 重新规划  │
    │   计划    │  │  或取消   │
    └─────┬─────┘  └──────────┘
          ↓
    ┌───────────┐
    │ 5. 返回   │
    │   结果    │
    └───────────┘
```

## 示例场景

### 场景1: 单步骤查询

**用户**: "gcc_002项目空间下有多少张表？"

**系统响应**:
```
📋 **执行计划**

我需要查询 DataWorks 系统来获取 gcc_002 项目中的表信息。

**步骤：**

1. 列出 gcc_002 项目中的所有表
   - 工具: ::dataworks.ListTables{"projectName":"gcc_002"}
   - 目的: 获取完整的表列表

**预期结果：** 您将知道该项目中确切的表数量和名称。

---

⚠️ **需要您的确认：**

回复 `yes` 或 `确认` 开始执行此计划
回复 `no` 或 `取消` 取消执行

💡 提示：你也可以修改问题后重新发送
```

**用户确认**: `yes`

**执行结果**:
```
**步骤 1:** 执行 `dataworks.ListTables`
{
  "Tables": ["users", "orders", "products", ... 共23张表]
}

---

**最终答案：** gcc_002 项目空间下共有 **23 张表**。
```

### 场景2: 多步骤任务

**用户**: "告诉我关于 gcc_002 项目的信息及其表结构"

**系统响应**:
```
📋 **执行计划**

我需要先确认项目存在，然后获取项目信息和表列表。

**步骤：**

1. 列出所有项目以确认 gcc_002 存在
   - 工具: ::dataworks.ListProjects{}
   - 目的: 确认项目存在并获取基本信息

2. 获取 gcc_002 项目的表列表
   - 工具: ::dataworks.ListTables{"projectName":"gcc_002"}
   - 目的: 获取所有表的详细信息

**预期结果：** 您将获得项目的完整信息和表结构概览。

---

⚠️ **需要您的确认：**

回复 `yes` 或 `确认` 开始执行此计划
回复 `no` 或 `取消` 取消执行
```

## 交互命令

### 确认命令
- `yes` / `y` - 确认执行
- `确认` / `执行` - 中文确认
- `confirm` / `ok` / `continue` - 英文确认

### 取消命令
- `no` / `n` - 取消执行
- `取消` / `否` - 中文取消
- `cancel` / `stop` - 英文取消

### 模式切换命令
- `/plan_mode` - 启用计划确认模式（默认）
- `/auto_mode` - 启用自动执行模式（无需确认）
- `/confirm_mode` - 查看当前模式状态

## 配置选项

### 启用/禁用确认模式

```python
# 在 ChatServer 初始化时
server = ChatServer(
    host="0.0.0.0",
    port=8765,
    llm_service=llm_service
)

# 默认启用确认模式
server.require_confirmation = True

# 运行时切换
server.require_confirmation = False  # 切换到自动模式
```

## 核心优势

### 1. 🔒 安全性
- 用户在执行前看到完整计划
- 避免意外执行危险操作
- 可以随时取消

### 2. 🎯 透明性
- 完全了解系统将要做什么
- 看到每一步的执行结果
- 清晰的推理过程

### 3. 🎮 控制权
- 用户决定是否执行
- 可以拒绝不合理的计划
- 主动权在用户手中

### 4. 🧠 智能化
- LLM 自动规划最优路径
- 一次性展示所有步骤
- 无需迭代次数限制

### 5. 📊 可视化
- 结构化的计划展示
- 清晰的执行步骤
- 易于理解的输出格式

## 使用建议

### 适合使用确认模式的场景
- 生产环境
- 涉及数据修改的操作
- 不熟悉的查询
- 需要精确控制的任务

### 适合使用自动模式的场景
- 开发/测试环境
- 只读操作
- 熟悉的查询
- 批量处理任务

## 技术实现

### 核心组件

1. **计划缓存** (`pending_plans`)
   - 保存待执行的计划
   - 支持超时清理

2. **确认机制** (`_create_and_present_plan`)
   - 提取工具调用
   - 生成确认消息
   - 等待用户响应

3. **执行逻辑** (`_execute_confirmed_plan`)
   - 按步骤执行工具
   - 收集执行结果
   - 生成最终答案

4. **模式切换** (`require_confirmation`)
   - 运行时切换
   - 状态持久化

## 常见问题

**Q: 如果用户长时间不确认怎么办？**
A: 计划会保存在 `pending_plans` 中，可以设置超时清理机制。

**Q: 能否支持修改计划而不是简单确认/取消？**
A: 当前版本不支持，但可以在后续版本中添加。

**Q: 自动模式和确认模式可以混用吗？**
A: 可以，通过命令随时切换。

**Q: 计划执行失败会怎样？**
A: 系统会显示错误信息，已执行的步骤结果会被保留。
